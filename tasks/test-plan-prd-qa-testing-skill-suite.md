# Test Plan: QA Testing Skill Suite (Ralph-Pattern)

## Source
- Input mode: PRD mode
- Source PRD: `tasks/prd-qa-testing-skill-suite.md`
- Generated by: `skills/qa-plan-generator/SKILL.md`
- Generated artifact: `tasks/test-plan-prd-qa-testing-skill-suite.md`

## Scope
Validate the QA skill suite end-to-end for:
- Skill 1 (`qa-plan-generator`) PRD/objective-to-markdown generation behavior
- Skill 2 (`qa-plan-json`) markdown-to-JSON conversion and validation behavior
- Skill 3 (`qa-codex-loop`) JSON execution behavior, gate semantics, loop controls, and output artifacts
- Ralph-pattern alignment (naming, handoff flow, operational ergonomics)

Primary in-scope checks:
- Deterministic output structures
- Deterministic ordering and ID generation
- Validation/error reporting quality
- Machine-readable PASS/FAIL outcomes
- Loop boundedness and fail-fast behavior

## Out of Scope
From PRD non-goals, excluded from this test plan:
- Full autonomous product implementation beyond test-related fixes
- Replacing CI systems (integration behavior only)
- Organizational ownership triage automation
- Building a custom test runner from scratch

## Assumptions and Ambiguities
- **A-001**: “Deterministic” means repeated runs with identical input produce byte-stable structure/order (minor timestamp/path variations allowed only where explicitly documented).
  - Assumed interpretation: section order, scenario order, and IDs are stable.
  - Impact if wrong: false failures in regression assertions.
- **A-002**: “Actionable validation errors” requires both reason + source location.
  - Assumed interpretation: errors include at least one file/line reference and remediation hint.
  - Impact if wrong: ambiguous triage and slower fix loops.
- **A-003**: Tool choice uses labels `codex` and `claude-code` exactly.
  - Assumed interpretation: parser rejects unknown values with typed error.
  - Impact if wrong: silent fallback could hide execution mismatches.
- **A-004**: “Affected tests” in remediation loop means tests related to changed files or explicitly failed test IDs.
  - Assumed interpretation: executor records rerun set explicitly in artifacts.
  - Impact if wrong: loop behavior becomes non-auditable.
- **A-005**: “Mirror Ralph ergonomics” is validated via naming, entrypoint behavior, log conventions, and handoff docs.
  - Assumed interpretation: parity with existing Ralph conventions, not identical implementation.
  - Impact if wrong: subjective acceptance disputes.

## Risk Areas
1. Non-deterministic markdown/JSON generation causing flaky plan conversion and review churn.
2. Partial traceability (US/AC not mapped) creating coverage blind spots.
3. Weak schema validation allowing malformed plans into executor.
4. Executor runs out of order, creating false PASS/FAIL outcomes.
5. Loop controls not enforced, causing runaway or premature termination.
6. Incomplete evidence artifacts making failures non-debuggable.
7. Divergence from Ralph conventions reducing maintainability and onboarding speed.

## Execution Strategy
1. **Static validation phase**
   - Lint markdown structure and naming/path contracts.
   - Validate converter schema and negative fixtures.
2. **Generator/converter integration phase**
   - PRD-mode and objective-mode generation tests.
   - Markdown-to-JSON conversion determinism tests.
3. **Executor phase**
   - Tool selection, ordering, PASS/FAIL output checks.
   - Controlled failure scenarios to validate loop semantics.
4. **Pattern-alignment review phase**
   - Manual checks for Ralph-style handoff/documentation/log ergonomics.
5. **Regression re-run**
   - Re-run P0 scenarios on final candidate.

Recommended cadence:
- P0 suite per commit touching skills or executor
- P1 suite before merge/release candidate

## Entry/Exit Criteria

**Entry Criteria**
- Skill files exist for all three QA skills.
- Required fixtures are present (valid and malformed markdown plans, execution fixtures).
- Local execution environment has required CLI dependencies.

**Exit Criteria (PASS)**
- All P0 scenarios pass.
- No open schema validation defects.
- Deterministic ordering/ID checks pass on repeated runs.
- Gate semantics (PASS only when all required tests pass) confirmed.
- Traceability section shows all US/AC/FR covered.

**Exit Criteria (FAIL)**
- Any P0 scenario fails.
- Unmapped US acceptance criteria or FR remain.
- Missing machine-readable status output or missing evidence artifacts.

## Evidence Requirements
For each executed scenario, collect:
- Command invocation and exit code
- Relevant stdout/stderr snippets
- Generated artifact paths and hashes (where determinism is asserted)
- For failures: root-cause notes and reproduction input fixture
- For loop tests: iteration timeline including affected/full reruns

Minimum artifact set:
- Generated markdown plan(s)
- Converted JSON plan(s)
- Executor run summary (per-test + aggregate)
- Validation error outputs for negative tests
